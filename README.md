# Mindbox SRE

## Что сделано

Сделан deployment веб-приложения в Kubernetes, упор на:
- отказоустойчивость
- адекватное потребление ресурсов
- предсказуемое масштабирование
- корректные обновления без даунтайма

---

## Как я рассуждал

В задании есть несколько важных вводных:
- приложение стартует 5–10 секунды
- первые запросы требуют больше CPU
- стабильное потребление около 0.1 CPU и около 128MB памяти
- 4 пода в пике
- ночью нагрузки меньше
- кластер мультизональный

От этого и строится весь написанный конфиг.

---

## Основные решения

### 1. Количество подов

- `minReplicas: 2` — базовый уровень
- `maxReplicas: 6` — запас относительно пика

Не держим 4 пода, потому что экономим ресурсы, масштабируемся по факту нагрузки

---

### 2. Масштабирование

Используется HPA по CPU.

CPU самый надёжный показатель, если у нас нет нужных метрик, по типу рпс или латенси

Параметры:
- `scale up` быстрый → чтобы не ловить деградацию на пике
- `scale down` медленный → чтобы избежать флаппинга

---

### 3. Ресурсы

- `requests.cpu = 100m` → соответствует реальным запросам
- `limits.cpu = 1000m` → даём запас на первые запросы

---

### 4. Пробы

Учитываем медленный старт:

- `startupProbe` → ждём запуска приложения
- `readinessProbe` → ждём возможности чтения
- `livenessProbe` → если повисло, перезапускаем

---

### 5. Обновления

- `maxUnavailable = 0`
- `maxSurge = 1`
Сделано так, чтобы не словить потерю доступности и минимальный расход ресурсов

---

### 6. Плавное завершение работы

- `preStop + sleep`
- `terminationGracePeriodSeconds`
Плавно выводим под, дабы случайно не обрезать уже обрабатываемые запросы

---

## Компромиссы

Какие упрощения я сделал:
- нет кастомных метрик (RPS, latency)
- нету балансировщика
- нет фиксированной версии образа (используется latest, в проде только конкретную версию)
- нет отдельной настройки автоскейла под зоны

---

## Решения компромиссов

Если убирать упрощения и делать всё в прод, то надо добавить:
- метрики приложения (например RPC)
- добавить Prometheus + Grafana
- настроить алертинг
- зафиксировать версии образов
---

## Вывод

Решение строится вокруг простых идей взятых из задания:
- не тратить ресурсы без необходимости
- быстро реагировать на рост нагрузки
- не терять доступность при сбоях и релизах

Без сильных сложностей, учитывая проблемы прода, но со своими упрощениями.
